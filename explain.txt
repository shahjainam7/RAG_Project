# How the Advanced RAG System Works

This project has been upgraded from a basic "Naive RAG" (Retrieval-Augmented Generation) pipeline into an Advanced RAG system. It is designed to take a user's competitive programming question, efficiently search through a database of ~2,900 LeetCode problems, and provide an expert AI-generated answer regarding the optimal approach and time complexity.

Here is a detailed, easy-to-understand breakdown of how the entire system works under the hood.

## 1. What Data is Used for Searching?
To find the right problem, the system doesn't just scan the entire text file blindly. It selectively extracts the most important parts to create an "embedding" (a mathematical representation of the text).

* **What gets embedded:** Only the **Problem Title** and the **Problem Description** are combined into a single string.
* **Why exclude the rest?**
  * **Boilerplate/Noise:** Words like "Constraints:" or "Hints:" appear in every problem. Including them confuses the search engine into thinking all problems are similar.
  * **Token Limits:** The AI models have memory limits (e.g., 256 tokens). By stripping out the noise, we ensure the core logic of the problem fits perfectly into memory without getting cut off.
* **What about the other data?** Information like "Topics" and "Expected Time Complexity" are still saved in the database as **Metadata**. We don't search by them, but once we find the relevant problem using the description, we pull this metadata to help formulate the final answer.

## 2. The Retrieval Pipeline (How we find the answer)
When a user asks a question, the system goes through a multi-step process to find the absolute best match:

### Step A: The Semantic Cache (The Shortcut)
Before doing any heavy lifting, the system checks its **Semantic Cache**. 
* If a user asks a question that is at least 90% similar to a question asked previously, the system instantly returns the saved answer from memory. 
* This bypasses the database and the LLM completely, making the system incredibly fast and cost-efficient.

### Step B: Hybrid Search (The Wide Net)
If the question is new, the system searches the Pinecone vector database using a **Hybrid Search** approach:
1. **Dense Search (Semantic Meaning):** Uses the `all-MiniLM-L6-v2` model to understand the *meaning* or *concept* behind the question (e.g., recognizing that "finding the shortest path" relates to graphs).
2. **Sparse Search (Exact Keywords):** Uses a `BM25Encoder` to look for exact keyword matches (e.g., specific terms like "Dijkstra" or "Segment Tree").
* The database combines these two scores (using a dot-product metric) to return the top 20 potential candidate problems.

### Step C: Reranking (The Quality Filter)
Fetching 20 problems is great for casting a wide net, but too much text to send to the final AI. 
* The system uses an advanced **Cross-Encoder model** (`ms-marco-MiniLM-L-6-v2`) to rigorously score and compare the user's exact query against those 20 candidates. 
* It filters out the "semantic noise" and narrows the list down to the absolute best **5 problems**.

## 3. The Final Output (The Coach)
Finally, those top 5 perfectly-matched problems (along with their metadata) are sent to an LLM (via OpenRouter/Gemini) acting as an expert competitive programming coach. 

The LLM analyzes the context and provides a concise, structured answer:
1. **OPTIMAL METHOD:** The primary algorithm or data structure needed.
2. **TIME COMPLEXITY:** The exact mathematical efficiency.
3. **WHY:** A one-sentence explanation of why this is the best approach for the given constraints.

## Summary
By combining **Smart Data Parsing**, **Semantic Caching**, **Hybrid (Sparse + Dense) Search**, and **Cross-Encoder Reranking**, this system ensures that the AI coach gets the fastest, most precise, and least noisy context possible to answer your programming questions!
